Working Security Case Study

Watch sessions, moderation, and audit trails move together.

WebShield Lab is a live Flask application for testing how account flows, admin review, and evidence capture behave under normal use.

Create an account, submit a support ticket, inspect the dashboard, and trace what lands in the audit log. When you need a contrast case, the demo area exposes intentionally unsafe routes in isolation instead of weakening the whole application.

Operational Trace

See which decisions changed the system.

Moderation actions, password changes, lab activity, and role updates are recorded as events you can review later instead of claims you have to explain from memory.

Secure Path

Keep normal user flows predictable.

Registration, login, ticket submission, and admin routes stay tied to sessions, validation, and ORM-backed data access so the secure behavior remains the default behavior.

Demo Isolation

Expose risky behaviour without trashing the main app.

The lab routes live behind `LAB_MODE=demo`, making it easier to capture contrast cases locally while keeping the primary workflow clean and defensible.

Suggested Walkthrough

Run the app like a real system, not a mock demo.

  1. 1
    Create a standard account and sign in.

    Start with the same registration and session flow a normal user would take.

  2. 2
    Submit a support ticket and inspect the dashboard.

    Watch the record appear in the user-facing workflow before any administrative action happens.

  3. 3
    Review the admin and audit views.

    Compare what changed, who changed it, and which events were written to the evidence trail.

  4. 4
    Switch to demo mode only when you need the contrast.

    Use the vulnerable routes to document before-and-after behaviour without blending the unsafe path into regular use.

Environment Split

Secure mode and demo mode should feel different.

Secure mode

Normal routes keep authorization checks, CSRF validation, audit recording, and safer database access in place. This is the environment for day-to-day development and evidence collection.

Demo mode

Intentional weaknesses are exposed only through the lab routes so you can demonstrate unsafe rendering, weaker access checks, or raw SQL behaviour without pretending the entire application is careless by default.
